# Phase 0 Research: Three.js Browser Network Topology Visualization

## 1. Delivery mechanism for a single, `file://`-safe HTML artifact

**Decision**: Vendor Three.js **r147** (`three@0.147.0`) specifically — its `build/three.js` (classic global/UMD build defining `window.THREE`) plus its same-release `examples/js/controls/OrbitControls.js` and `examples/js/loaders/GLTFLoader.js` (pre-ESM global addon builds) — as static files committed under `workspace/skills/threejs-network-viz/vendor/three/`. These are inlined as plain `<script>` tags (no `type="module"`) into the generated HTML.

**Rationale**: Three.js dropped the classic UMD core build (`build/three.js`) at r161 and dropped non-module addon builds (`examples/js/`, including `OrbitControls`/`GLTFLoader` as globals) even earlier, at r148 — confirmed directly against unpkg (`three@0.161.0/build/three.js` → 404; `three@0.147.0/build/three.js` → 200; `three@0.148.0/examples/js/controls/OrbitControls.js` → 404; `three@0.147.0/...` → 200). r147 is the last release where both exist and are guaranteed API-compatible with each other. Modern Three.js is ES-module-only, and ES module `import` is blocked by browsers when a page is opened via `file://` (null origin, CORS blocks the module fetch) — exactly the scenario this feature requires (double-click a generated HTML file, no server). Vendoring r147's global builds sidesteps that entirely with zero build step and zero CDN dependency at render time (the file stays self-contained even offline).

**Alternatives considered**:
- Bundle current Three.js via esbuild/rollup into one classic script at build time — adds a JS build toolchain dependency to a Python-skill project for a problem an older release already solves for free; rejected for unnecessary complexity.
- Use `<script type="importmap">` to resolve bare `three` imports — also fails on `file://` for the same origin/CORS reason; rejected.
- A third-party UMD-wrapper package (`@vladkrutenyuk/three-umd`) — low-activity, unreleased personal project; not suitable to depend on for a shipped feature.
- Accept the server-required fallback (Story 5's original secondary option) — rejected because r147 vendoring achieves the zero-server goal directly with no compromise.

**Trade-off accepted**: r147 (mid-2023) lacks a few years of newer Three.js features/perf work. This is acceptable — the feature only needs stable Group/Mesh/Line/Tube/GLTFLoader/OrbitControls primitives, all present and stable at r147.

## 2. Embedding real-stencil 3D models into the single HTML file

**Decision**: Any resolved glTF/GLB model is downloaded once by the skill (Python side, via `sketchfab-mcp`), base64-encoded, and written into the HTML as an inline string. At render time, the page's own JS decodes the base64 string back into an `ArrayBuffer`/`Uint8Array` and calls `GLTFLoader.parse(arrayBuffer, '', onLoad, onError)` — never `GLTFLoader.load(url, ...)`.

**Rationale**: `GLTFLoader.parse()`'s documented signature explicitly accepts a binary `ArrayBuffer` (a parsed `.glb`) in addition to a JSON string, performing zero network fetches; the `path` argument only matters for resolving *external* resources referenced by a non-binary/non-self-contained asset, which doesn't apply to a self-contained `.glb`. This is the mechanism that lets real-stencil mode honor FR-001/FR-019's single-file requirement instead of trading it away.

**Alternatives considered**: Reference the model as a separate `.glb` file next to the HTML — rejected outright per the Clarification session decision (real-stencil mode must still be a single file); ship models as data-URI `src` on an `<img>`-style tag — not applicable to binary glTF geometry, GLTFLoader's own parse path is the correct primitive.

## 3. Sketchfab license verification for real-stencil mode

**Decision**: `assets.py` calls `sketchfab-search` (via the vendored `sketchfab-mcp` server) with `downloadable=true` and a role-derived query, then for each candidate calls `sketchfab-model-details` and inspects the returned license object's `slug` field, accepting only `slug === "cc0"`. Any candidate with a missing, non-`cc0`, or unrecognized license slug is rejected and the resolver moves to the next candidate or falls back to procedural.

**Rationale (updated during implementation with live-verified facts — see below)**: Sketchfab's underlying Data API v3 does support server-side pre-filtering (`GET /v3/search?type=models&license=cc0`, confirmed live) and defines an explicit `cc0` license slug (among `by`, `by-sa`, `by-nd`, `by-nc`, `by-nc-sa`, `by-nc-nd`, `free-st`, `st`, `ed` — confirmed live against `GET /v3/licenses`). The vendored community `sketchfab-mcp-server`'s `sketchfab-search` tool exposes no `license` passthrough parameter, so per-candidate verification via `sketchfab-model-details` is the enforcement point, satisfying FR-019a.

**Correction to the original (pre-implementation) version of this research**: the original text above stated the model-details tool "does return full model metadata including license" as a reason not to touch the vendored server. That was wrong. Direct inspection of the vendored server's actual TypeScript source (`mcp-servers/sketchfab-mcp-server/index.ts`) during implementation showed its `formatModelForDisplay()` helper silently DROPPED the `license` field from the tool's text output entirely, even though the raw Sketchfab API response includes it — meaning FR-019a's verification step had nothing to read. This was fixed by patching `formatModelForDisplay()` to include `License: ${JSON.stringify(model.license)}` in its output (see the "NetClaw patch" comments in `index.ts`) and rebuilding (`npm run build`). Separately, live API calls also showed the TypeScript type `license?: string` was itself wrong — the real API returns `license` as an object (confirmed shape on `/v3/models/{uid}`: `{uri, label, fullName, requirements, url, slug}`, with the CC0 record's slug literally `"cc0"` and uid `7c23a1ba438d4306920229c12afcb5f9`) — also corrected in the patch.

**Alternatives considered (revised)**: The original research rejected patching the vendored server as "unnecessary maintenance burden," reasoning that the existing tool already fully satisfied the requirement. That premise turned out to be false once the real tool output was inspected — patching was in fact necessary, not optional, and the fix applied was the minimal one-line addition needed (not a broader fork), keeping the maintenance burden low as originally intended.

**Live findings during implementation** (informs `assets.py`'s fallback behavior, not just its happy path): searching Sketchfab for network-gear terms ("router", "server rack", "electronic box") filtered to `license=cc0` returned zero or entirely irrelevant results in practice — procedural-shape fallback is the COMMON case for real network-device roles, not a rare edge case, exactly as anticipated in the original asset-pipeline design memo. Additionally, not every downloadable Sketchfab model has a ready-made glTF/GLB export — `sketchfab-download` requesting `format: "glb"` silently substitutes a different format (source/gltf/usdz) when GLB isn't available for that specific model, which `assets.py`'s `download_and_embed()` must detect and treat as a fallback condition (checking the tool's response text for the exact "in glb format." success phrasing, not just success generally). Both the CC0-license and CC0-with-GLB-available full pipeline (search → verify → download → base64-embed → `GLTFLoader.parse()` in a real headless browser) were exercised end-to-end with a real Sketchfab model during implementation and rendered with zero errors.

## 4. Reuse vs. re-derivation of topology layout and device-type inference

**Decision**: `layout.py` and `materials.py` in the new skill port and adapt (not blindly reinvent) the proven algorithms already in `workspace/skills/ue5-network-viz/layout.py` (`ForceDirectedLayout`, including its centroid re-centering fix) and `materials.py` (`infer_device_type`, `DEVICE_TYPE_COLORS`) — copied into this skill's own modules rather than imported cross-package.

**Rationale**: `ue5-network-viz/layout.py`'s `ForceDirectedLayout` is already engine-agnostic (pure Python, produces abstract `Vector3` positions with no UE5 API calls) and already carries the centroid-centering fix from commit `5281cac` that this spec's FR-008 exists specifically to avoid re-breaking. However, this repo's skills are not structured as an importable shared package today (relative imports with `try/except ImportError` sys.path fallbacks inside each skill directory, per `ue5-network-viz/renderer.py`), so a live cross-skill-directory import would be fragile and outside this feature's scope to fix. Porting the already-correct logic avoids re-deriving the same scale/centering bugs from zero, while not taking on a cross-skill architecture refactor this feature doesn't need.

**Alternatives considered**: Extract a genuinely shared `netclaw_topology` library now — correct long-term direction, but a larger refactor touching 024/044/045's existing code, out of scope for this feature; noted as a reasonable future follow-up, not blocking.

## 5. Testing approach

**Decision**: Pure-Python core logic (topology modeling, layout/centering, color mapping, asset-resolution/fallback) gets full unit-test coverage with `sketchfab-mcp` calls mocked. Topology-source composition (Story 2) gets a smaller set of live integration tests against at least one real, already-running topology-source MCP, asserting on the structure of the generated file rather than pixel-level rendering.

**Rationale**: Unlike UE5/Blender (no unit-testable surface independent of a running desktop engine), this feature's core is pure Python with no live dependency — the primary verification layer should be fast, deterministic unit tests. Sketchfab-catalog content changes over time and is rate-limited, so mocking it for correctness tests (CC0 acceptance/rejection logic, fallback behavior) avoids flakiness while still testing the real decision logic; only the topology-source integrations need a live test, mirroring the existing "never mock the dependency whose failure would matter most" rule from 044/045 (there: a live UE5 instance; here: a live topology source, since a scene built from bad/misread topology data would be the most damaging silent failure).

**Alternatives considered**: Full live Sketchfab integration tests — rejected due to catalog volatility/rate limits, which would make CI flaky for reasons unrelated to code correctness.
