# Custom Installer for MCP Servers with DefenseClaw Production Mode

I want to make a custom installer for every MCP server as an option to install or not and have this be able to be installed via docker compose, with Defense claw with production mode — enforced, honest, and durable (feature 057).

N2N_RISK_MODE=production enforces, fail-closed, rather than merely flagging. Layered, production-grade security — and posture that never lies:

1. Member sandbox — host-level kernel confinement. Each member runs as a hardened systemd unit (NoNewPrivileges, ProtectSystem=strict, the operator's master .env made inaccessible, private /tmp, and syscall/namespace limits on native Linux); on-demand members cold-start in a transient hardened systemd-run unit. The member keeps its real tools/network but is kernel-confined. (OpenShell containers were evaluated and rejected for live-infra members — empty, egress-denied sandboxes; host confinement is what actually works.)
2. Model-guard — DefenseClaw LLM guardrail proxy. Model I/O routes through the DefenseClaw Go proxy (:4000) for prompt/response inspection; member skill/MCP sets are DefenseClaw component-scanned before they run. If the guard is unavailable, delegations fail closed (no silent bypass to a direct provider).
3. Audit — GAIT immutable git trail. Every delegation/enrollment/removal/quarantine is committed to ~/.openclaw/n2n/gait/ (append-only) on both Border and member sides, cross-referenced to the SQLite audit.
4. Least-privilege secrets by construction — each member gets only its integration's .env slice; the sandbox hides the master .env.
5. Honest posture — testing / production — enforced / production — DEGRADED (<controls> missing); never a false enforced claim. Per-control policy: containment gap (sandbox / model-guard) blocks; audit-only gap (GAIT) runs flagged audit-degraded; N2N_STRICT_ALL=1 blocks on any gap.
6. Durable runtime — mesh daemon + always-on members as systemd --user services (Restart=always), generated by scripts/in2n-services.py; fault isolation distinguishes daemon vs member vs backend.
7. A2A cards advertise the risk's posture and LLM capability (model + guarded flag) so federated peers see a neighbour's security + reasoning tier.
